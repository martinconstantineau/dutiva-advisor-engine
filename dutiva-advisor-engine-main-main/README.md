# dutiva-advisor-engine

A headless Qwen-powered Canadian HR compliance advisor engine with French-ready advisor contract and routing foundations, with production French guidance validation still in progress. It produces structured advisor responses with explicit rendering gates for the consuming application.

This is not a chatbot UI. It is a production-grade advisor pipeline that routes, retrieves, validates, and composes HR compliance guidance for integration with the main Dutiva application or any compatible consuming app.

---

## Purpose

`dutiva-advisor-engine` exposes a single HTTP endpoint that accepts a user message and context, then returns:

- A natural `conversationalResponse` string for the main chat surface.
- An optional structured `workspace` payload for HR compliance UI cards.
- A `route` object that tells the consuming app exactly what it is allowed to render.
- `jurisdiction`, `risk`, `professionalReview`, and `quality` fields for downstream logic.

**The consuming app must obey `route` rendering gates.** The main chat surface renders only `conversationalResponse`. The structured workspace UI renders only `workspace`. If a gate is false, the related UI and content must not be returned or rendered.

## Supported Jurisdictions

| Jurisdiction | Status | Guidance depth |
| --- | --- | --- |
| Federal (Canada Labour Code, CHRA) | Supported | **Deep** — section-level, citation-linked provisions generated by the automated `advisor-training` pipeline |
| Ontario | Supported | **Curated** — hand-written ESA / OHSA / Human Rights Code entries spanning termination, harassment, accommodation, leave, reprisal, safety, pay/hours, and medical disclosure; automated statute ingestion not yet built |
| Québec | Supported | **Curated** — hand-written ARLS / Charter / LSST entries across the same topics (incl. pay/hours and medical disclosure); automated statute ingestion not yet built |
| British Columbia | Coming soon | — |
| Alberta | Coming soon | — |
| All other provinces/territories | Not yet available | — |

**Coverage depth varies by jurisdiction.** Federal questions are backed by the full automated pipeline (section-cited Canada Labour Code, Canada Labour Standards Regulations, and Canadian Human Rights Act provisions — thousands of records). Ontario and Québec support currently rests on a small set of hand-curated guidance entries in `src/retrieval/retrieveGuidance.ts`; automated provincial statute ingestion is scaffolded (`advisor-training/canada/acts/ontario`, `.../quebec`) but not yet populated. A federal query therefore draws on far more granular legal basis than an Ontario or Québec query. The federal harassment/violence topic (Bill C-65, SOR/2020-130) is itself still on the curated path pending pipeline ingestion.

**Jurisdiction is never silently assumed.** If `province`, `companyContext.province`, `userContext.preferredProvince`, and conversation history all fail to establish a province, and `isFederallyRegulated` is unknown, then `jurisdiction.status` is `"unknown"` and `jurisdiction.isFederallyRegulated` is `null`. Guidance is jurisdiction-neutral and no province-specific or federal legal citations are returned as authoritative.

## Mode Naming

Two separate mode concepts exist:

**`RequestedAdvisorMode`** — the client hint sent in the request:

| Value | Description |
| --- | --- |
| `"hr_compliance"` | General HR compliance (default) |
| `"legal_issue_spotting"` | Flag legal issues |
| `"document_drafting"` | Generate a draft document |
| `"supportive_triage"` | Supportive/wellness mode |

**`AdvisorResponseMode`** — the actual internal mode set by the deterministic router:

| Value | Description |
| --- | --- |
| `"hr_compliance_advisor"` | General HR compliance guidance |
| `"high_risk_escalation"` | High-risk HR route (harassment, wrongful dismissal, etc.) |
| `"supportive_triage"` | Personal wellness, crisis, or supportive guidance |
| `"document_drafting"` | Document generation |
| `"legal_issue_spotting"` | Legal issue identification |
| `"out_of_scope_redirect"` | Out-of-scope topic redirect |

The request `mode` is a hint only. The deterministic router overrides it for crisis, personal wellness, out-of-scope, and high-risk routes. The response exposes the actual `route.responseMode`.

**Default:** `mode` defaults to `"hr_compliance"` in the request. Do not send `"hr_compliance_advisor"` as a request mode — that is an internal response mode.

## Routing Matrix

Routing happens deterministically before LLM generation and before retrieval. The router sets all gates.

The `employee_medical_or_accommodation` route matches accommodation, disability, and medical-restriction queries regardless of whether mental-health language is present. Matched patterns include: `asked for accommodation`, `requested accommodation`, `needs accommodation`, `disability accommodation`, `duty to accommodate`, `medical condition`, `medical restrictions`, `functional limitations`, `functional abilities`, `doctor's note`, `doctor's notes`, `physician's note`, `medical note`, `medical certificate`, `medical documentation`, `medical information`, `functional capacity form`, `return-to-work note`, `fit-to-work note`, `what medical information can we ask for`, `employee gave us a doctor's note`, `return to work restrictions`, `modified duties`, `undue hardship`, `human rights accommodation`, and French equivalents: `accommodement`, `demande d'accommodement`, `mesure d'adaptation`, `handicap`, `limitations fonctionnelles`, `restrictions fonctionnelles`, `capacités fonctionnelles`, `formulaire de capacités fonctionnelles`, `billet médical`, `note médicale`, `certificat médical`, `documentation médicale`, `renseignements médicaux`, `information médicale`, `aptitude au travail`, `billet de retour au travail`, `restrictions médicales`, `contrainte excessive`, `droits de la personne`. These queries are routed to `employee_medical_or_accommodation` (not `general_hr_compliance`), and the fallback response focuses on confidentiality, functional information, accommodation and leave options, documentation, and anti-reprisal — **not** termination or severance facts.

The `leave_or_absence` route now matches federal/CLC leave queries, not only named leave types. Matched patterns include: `leave rules`, `leave entitlements`, `leave provisions`, `federal leave`, `federally regulated leave`, `Canada Labour Code leave`, `CLC leave`, `family responsibility leave`, `personal leave`, `protected leave`, and French equivalents: `règles de congé`, `droits aux congés`, `congé de maternité`, `congé parental`, `congé de maladie`, `congés protégés`, `Code canadien du travail congé`. A query like "Current Canada Labour Code leave rules" routes to `leave_or_absence`, not `general_hr_compliance`.

The `termination_or_discipline` route matches on termination and discipline keywords in both English and French, including: `terminated`, `dismissed`, `fired`, `let go`, `constructive dismissal`, `wrongful dismissal`, `discipline`, `notice period`, `termination notice`, `pay in lieu`, `severance package`, `without cause`, as well as French equivalents: `préavis`, `congédiement`, `licenciement`, `indemnité de départ`, `mise à pied`, `congé sans cause`.

**Fallback topic scoping:** Deterministic fallback responses (used when no LLM key is configured) are now scoped to the route topic. Accommodation fallbacks ask for accommodation-relevant facts (jurisdiction, what support was requested, functional restrictions, safety concerns) — not termination or severance facts. Harassment fallbacks ask for complaint, parties, and safety facts. Leave fallbacks ask for the type of leave and jurisdictional context. Only the `termination_or_discipline` route fallback asks for length of service, employment contract terms, and severance offer details.

## Route Rendering Gates

The consuming app **must** check these before rendering any structured content:

| Field | What it blocks |
| --- | --- |
| `route.workspaceAllowed === false` | Workspace payload, all compliance cards |
| `route.legalBasisAllowed === false` | Legal citations, statute references |
| `route.retrievalAllowed === false` | Retrieval match labels, guidance sources |
| `route.suggestedDocumentsAllowed === false` | Suggested documents |
| `route.webSearchAllowed === false` | Real-time web sources (`webSearch` field) |

`route.legalBasisAllowed` is a **final effective rendering gate**, not an abstract eligibility flag. It is `true` only when the topic is eligible, the jurisdiction is known or explicitly assumed, there are no unresolved context conflicts, and the feature is enabled. If jurisdiction is unknown, conflicted, or unsupported, the engine sets this gate to `false` and omits `workspace.legalBasis`.

`route.retrievalAllowed` is also a final rendering gate. **The consuming app must only render `workspace.retrievedGuidance` when this is `true`.** Four layers of filtering apply to public `workspace.retrievedGuidance`:

1. **Unknown/conflicted jurisdiction:** Only jurisdiction-neutral (`ALL`) items are shown. Province-specific and federal-only items are withheld with a quality warning: `"Jurisdiction-specific retrieved guidance was withheld until jurisdiction is confirmed."`
2. **Known provincial jurisdiction (e.g. `ON`):** Only `ALL` items and items matching the known province are shown. Items from other provinces (e.g. `QC` items when province is `ON`) and federal-only items (`federalOnly: true`) are withheld with a quality warning.
3. **Known federal jurisdiction:** Only `ALL` items and `FEDERAL`-province items are shown. Provincial items are withheld.
4. **Topic-alignment filter:** After jurisdiction filtering, items whose guidance category does not match the query topic are withheld. The effective topic set is resolved in two steps: (a) the query text is scanned for recognised HR sub-topic terms (accommodation, medical restrictions, modified duties, functional limitations, doctor's note, harassment, termination, leave, reprisal, workplace safety, compensation, and French equivalents); (b) if no topic terms are found in the text but the route intent is specific (e.g. `employee_medical_or_accommodation`), the intent is used as a fallback to derive allowed categories. A harassment query therefore returns only harassment-category items; a medical-restrictions/modified-duties query returns only accommodation and medical-disclosure items; termination, leave, reprisal, and safety items are suppressed even if they matched the jurisdiction and scored above the retrieval threshold. A quality warning is added: `"Retrieved guidance for unrelated topics was withheld (topic-alignment filter)."` This filter is not applied only when the intent is `general_hr_compliance` and no specific topic terms are detected — i.e. a genuinely broad query. **Public `workspace.retrievedGuidance` must pass both jurisdiction filtering and topic/intent alignment. The engine does not display guidance merely because it matches the jurisdiction.**

    Cross-category allowances: `accommodation` and `medical_disclosure` are always allowed together (tightly related). All other categories are mutually exclusive unless their keywords appear in the query text.

When `route.retrievalAllowed === false` (e.g. current-info short-circuit, personal/crisis route), no retrieval results appear at all.

**Public `workspace.legalBasis` contains only citations with `validationStatus: "valid"`.** Raw LLM-generated citations that are unvetted (`requires_review`) are never placed in the public `workspace.legalBasis`. They are surfaced in `quality.warnings` and, when `options.includeDebug === true`, in `debug.unvettedCitations`. The consuming app does not need to filter `requires_review` items — they will not appear in the public payload.

Personal wellness, crisis, ambiguous, out-of-scope, and document-drafting routes set all five gates — including `webSearchAllowed` — to `false`. The workspace payload will be absent. `route.webSearchAllowed` is a **final effective rendering gate**, not merely an intent-level eligibility flag. It is `true` only when ALL of the following hold for this specific response: the intent is HR/compliance-eligible; `options.enableWebSearch` was `true` in the request; `WEB_SEARCH_ENABLED=true` globally; the Startpage endpoint and API key are configured; and the query requires current or external information. If global config or the per-request option disables web search, `route.webSearchAllowed` is `false` even for HR-eligible topics. If the query does not require current/external info, `route.webSearchAllowed` is `false`. The public `webSearch` field is present only when `route.webSearchAllowed` is `true` and web search actually ran. The consuming app must **never carry `webSearch` results from a prior turn** when the current turn's `webSearchAllowed` is `false`. Prior retrieval, workspace, or web search data from a previous turn must never be rendered when the current turn's gates disable it.

## Jurisdiction Handling

### Province defaults to `null`

`province` defaults to `null` — never Ontario. The engine will not cite Ontario law unless `province: "ON"` is explicitly provided. All deprecated helpers that previously fell back to Ontario have been removed; an unknown province always stays `null`.

### Generic "Canada" does not imply federal jurisdiction

Generic words like "Canada" or "Canadian" alone do not set the jurisdiction to federal. The engine only treats explicit federal signals as federal: `federal`, `federally regulated`, `Canada Labour Code`, `CLC`, `federal jurisdiction`. If the user only says "Canada" and no province or federal status is provided, `jurisdiction.status` remains `"unknown"` and no province-specific or federal citations are returned as authoritative.

### Federal jurisdiction defaults to `null` (unknown)

`isFederallyRegulated` defaults to `null` (unknown):

| Value | Meaning |
| --- | --- |
| `true` | Known federally regulated — Canada Labour Code applies |
| `false` | Known not federally regulated |
| `null` / omitted | Unknown — do not assume |

When federal status is unknown and no province is given, `jurisdiction.status` is `"unknown"` and guidance is jurisdiction-neutral.

### Context precedence

When province or federal status can be inferred from multiple fields, the engine resolves them in this order:

1. Explicit request fields (`province`, `employerProvince`, `isFederallyRegulated`, `employeeCount`, `employerContext`)
2. `companyContext` fields (`companyContext.province`, `companyContext.federalJurisdiction`, `companyContext.employeeCount`, `companyContext.name`, `companyContext.industry`)
3. `userContext` preferences (`userContext.preferredProvince`, `userContext.role`)
4. Conversation history — only explicit, parseable province strings from prior user turns
5. Unknown / null

### Conflict detection

If overlapping fields conflict (e.g., `province: "ON"` but `companyContext.province: "QC"`), the engine does not silently resolve the conflict:

- `jurisdiction.notes` includes a plain-English conflict description.
- `quality.warnings` includes a conflict warning.
- `route.legalBasisAllowed` is set to `false` for the final response.
- `workspace.legalBasis` is omitted.
- No province-specific or federal legal conclusion is returned as authoritative until the conflict is resolved.

### Remote-work jurisdiction

For remote-work scenarios, the engine identifies possible jurisdictional issues, flags missing facts, and avoids definitive conclusions where governing law may depend on the employment relationship, workplace location, employer operations, and applicable legislation.

## Crisis Handling

Crisis language (self-harm, imminent danger, threats of violence) is intercepted before the LLM. The response is:

```json
{
  "isCrisis": true,
  "route": {
    "intent": "possible_crisis_or_self_harm",
    "responseMode": "supportive_triage",
    "surface": "advisor_chat",
    "retrievalAllowed": false,
    "workspaceAllowed": false,
    "legalBasisAllowed": false,
    "suggestedDocumentsAllowed": false,
    "webSearchAllowed": false
  },
  "risk": { "compliance": "low", "safety": "critical" },
  "professionalReview": { "recommended": true, "type": "emergency" }
}
```

Crisis responses do **not** include HR guidance, citations, retrieval cards, legal basis, suggested documents, or a workspace payload. Crisis handling cannot be overridden by the `mode` field.

Crisis resources are maintained in `src/safety/crisisRouting.ts` from public Canadian government sources. The LLM never generates phone numbers or URLs.

## Personal Wellness Routing

Inputs like "What self-care strategies could you recommend for me?" or "I feel depressed" route to `supportive_triage` with all workspace gates set to `false`. The engine:

- Provides a gentle, supportive response.
- Does not generate HR retrieval, legal basis, workspace cards, citations, or confidence labels.
- Does not assume a workplace or company context.
- Bridges softly to HR compliance if the user then provides workplace context.

"I feel depressed" is routed as `personal_mental_health` and the engine asks whether the situation is personal, employee-related, or employer-response context before providing HR guidance.

---

## Workplace Mental Health Routing

Inputs like "An employee told me they are depressed" or "My employee needs mental health leave" route to `employee_medical_or_accommodation` with workspace enabled. The engine surfaces:

- Sensitive medical and disability-related disclosure handling.
- Confidentiality and privacy obligations.
- No diagnosis or treatment advice.
- Duty to accommodate and leave considerations.
- Immediate safety assessment.
- Factual documentation guidance.
- Anti-reprisal and non-discrimination obligations.
- Jurisdiction confirmation if missing.

---

## Professional Review

`professionalReview` replaces the legacy `recommendLawyer: boolean` field. The engine recommends the right type of review based on route, risk, and facts:

| Scenario | `type` |
| --- | --- |
| Crisis / imminent danger | `emergency` |
| Harassment, wrongful dismissal, reprisal | `legal` |
| Unionized discipline | `union` or `legal` |
| Medical / accommodation complexity | `hr` or `medical` |
| Persistent personal wellness issue | `medical` where appropriate |
| Low-risk HR admin | `none` |

The engine does not always recommend a lawyer.

---

## Risk Model

Risk is split into two independent dimensions:

- `risk.compliance` — HR/employment law compliance risk: `low | medium | high | critical`
- `risk.safety` — personal safety risk: `none | watch | urgent | critical`

A crisis can be `safety: critical` without being an HR compliance matter. A termination can be `compliance: high` without being a personal safety risk.

## Legal Citations

Legal citations are optional and gated. They appear only when:

- The query is HR/workplace-related.
- `route.legalBasisAllowed === true`.
- Jurisdiction is known or explicitly assumed.
- The citation comes from vetted guidance data.
- Citation validation passes.

Citation validation statuses:

| Status | Where it appears | Rendering rule |
| --- | --- | --- |
| `valid` | `workspace.legalBasis` | May render as authoritative |
| `requires_review` | `quality.warnings`, `debug.unvettedCitations` only | Never in public legalBasis; do not render as authoritative |
| `suppressed` | `quality.warnings` only | Never rendered |

Raw LLM-generated citations (those that are unvetted or could not be matched to a known statute) are **withheld from the public `workspace.legalBasis`**. They appear only in `quality.warnings` (as a brief notice) and in `debug.unvettedCitations` when `options.includeDebug === true`. The consuming app does not need to filter them — they will not appear in the public payload.

Malformed citations (e.g. `s. (3)`, `An Act to consolidate certain statutes respecting labour, s. (4)`) are suppressed. The engine never invents replacement citations.

No plain-text field returned by the engine contains raw `**` Markdown.

## Getting Started

### Prerequisites

- Node.js >= 18.0.0
- A Qwen API key for live LLM generation (optional — deterministic fallback responses cover all regression cases including crisis, personal wellness, harassment, and employee medical/accommodation)

### Installation

```bash
git clone https://github.com/your-org/dutiva-advisor-engine.git
cd dutiva-advisor-engine
npm install
```

### Configuration

```bash
cp .env.example .env
```

```bash
QWEN_API_KEY=
QWEN_BASE_URL=
QWEN_MODEL=
QWEN_TIMEOUT_MS=30000
QWEN_MAX_RETRIES=2
PORT=3000
```

`QWEN_BASE_URL` is optional. When omitted, the provider uses the safe default `https://dashscope.aliyuncs.com/compatible-mode/v1` (DashScope/Qwen OpenAI-compatible endpoint). The provider will never fall back to the OpenAI API endpoint. The engine uses the provider-neutral interface from `src/llm/provider.ts`; the Qwen-specific implementation is in `src/llm/qwenProvider.ts`.

Provider errors are sanitized before they reach logs or API responses: API keys, bearer tokens, authorization headers, and credential-like strings are redacted so secrets cannot leak.

If `QWEN_API_KEY` is not configured, the engine still returns deterministic responses for routes that have fallback coverage (crisis, personal wellness, out-of-scope, ambiguous, and all HR intents). This is intentional for local testing and graceful degradation; it does not produce noisy warning logs.

The deterministic fallback is **jurisdiction-aware**: if jurisdiction is already known from the request context, the fallback response uses it (e.g. mentions Ontario ESA or Canada Labour Code) and does not ask the caller to provide province information that was already supplied. When jurisdiction is unknown, the fallback continues to ask for province or federal status. If relevant retrieved guidance is available, the fallback uses the top matching item's title and content to give a more helpful answer, without inventing citations.

### Running

Development:

```bash
npm run dev
```

Production:

```bash
npm run build
npm start
```

## French / Bilingual Support Status

The advisor engine supports `locale: "en" | "fr"` at the request/response-contract level and includes French-aware routing patterns for common HR topics such as accommodation, medical notes, harassment, leave, workplace safety, wellbeing, and crisis-sensitive routing.

However, the validated generated guidance corpus is currently English-only.

Current French support should be understood as foundation-level support, not full production-grade bilingual parity:

| Layer | French Status |
| --- | --- |
| API locale contract | Supported through `locale: "fr"` |
| LLM response language | Supported through prompt instructions |
| French-aware routing keywords | Partially supported |
| French crisis/wellbeing wording | Partially supported |
| French terminology helpers | Present |
| Generated French guidance corpus | Not validated / not user-facing |
| French source-law ingestion | Not implemented |
| Full French evaluation parity | Not complete |
| Production-grade reviewed French advisor content | Not complete |

Generated guidance cards may contain French placeholder fields such as `advisor_answer_fr_placeholder`. These placeholder fields are intentionally not exposed to users and must remain gated from runtime responses, workspace output, legal basis, citations, debug output, and prompt context.

In practical terms:

- English runtime guidance is active through the curated fallback guidance and the generated federal source-law corpus.
- French requests can be routed and answered in French by the LLM.
- The underlying validated retrieval corpus remains English-first.
- Full production French support requires a reviewed French guidance layer, French source-law ingestion where appropriate, and French-specific evaluation coverage.

Route enums and internal metadata remain stable English values regardless of locale. French employment law terms (harcèlement, préavis, congé, etc.) are matched correctly in retrieval and routing.

## Integration with the Main Dutiva App

1. Call `POST /api/advisor/respond` with the user message and context.
2. Send `mode: "hr_compliance"` (the default). Do not send `"hr_compliance_advisor"`.
3. Render `conversationalResponse` in the chat surface — it is clean prose with no raw Markdown.
4. Check `route.workspaceAllowed` before rendering any workspace cards.
5. Check `route.legalBasisAllowed` before rendering any legal citations.
6. Check `isCrisis` — if true, surface the crisis resources from `conversationalResponse` prominently. Do not show HR compliance content.
7. Check `jurisdiction.status` — surface a prompt to collect province or federal status when `"unknown"`.
8. Check `quality.warnings` — surface jurisdiction conflict warnings to the user when present.
9. Use `professionalReview` to surface the appropriate type of professional referral (not just a lawyer recommendation).
10. Use `risk.safety` to determine whether to surface immediate safety escalation UI.
11. Check `route.webSearchAllowed` before rendering any `webSearch` results. Do not carry `webSearch` results from a prior turn when the current turn's `webSearchAllowed` is `false`.
12. On a new turn: **never carry workspace, retrieval, legal, or web search data from a prior turn** when the new turn's gates are false. Build a fresh context per turn.

## Real-Time Web Search

The engine integrates **Startpage** as an optional real-time web-search discovery provider.

**Important distinctions:**

- **Qwen** is the LLM provider. It generates all conversational and workspace responses.
- **Startpage** is the web-search/discovery provider only. It is used to discover candidate source URLs for current HR/legal information.
- Startpage is **not** a legal authority. Startpage URLs are **not** returned as canonical citations.
- The engine validates, classifies, and returns the **underlying source page URLs** — never Startpage anonymous/proxy URLs.
- Web search does **not** replace vetted internal guidance retrieval. Both run separately and are kept distinct.

### Startpage Configuration

Web search is **disabled by default** (safe for tests and dev environments). Enable it explicitly:

```bash
WEB_SEARCH_ENABLED=true
WEB_SEARCH_PROVIDER=startpage

# Contracted/approved Startpage API adapter endpoint (required for live calls)
STARTPAGE_BASE_URL=https://your-startpage-adapter-endpoint.example.com
STARTPAGE_API_KEY=your_startpage_api_key

# Optional tuning
STARTPAGE_TIMEOUT_MS=10000
STARTPAGE_MAX_RESULTS=5
STARTPAGE_REGION=ca
STARTPAGE_LANGUAGE=en
WEB_SEARCH_CACHE_TTL_SECONDS=900
WEB_FETCH_TIMEOUT_MS=10000
```

Because Startpage does not publish an official public API, the provider is implemented as a configurable **adapter interface**. Configure `STARTPAGE_BASE_URL` to point to a contracted or approved API/proxy endpoint. The engine does **not** scrape Startpage HTML. If the endpoint is not configured, web search fails safely with a quality warning — it never throws a public API error.

No live Startpage network calls are made in tests (`WEB_SEARCH_ENABLED` defaults to `false`).

### Web Search Routing

Web search may run only when **all** of these are true:

1. `WEB_SEARCH_ENABLED=true` (global environment flag)
2. `options.enableWebSearch=true` (per-request flag — default `false`)
3. `route.webSearchAllowed=true` (set by the deterministic router)
4. The query requires current or external information (`requiresCurrentInfo` logic)

**Queries that trigger web search when enabled:**

- "What changed in Ontario employment law this year?"
- "Latest ESA update Ontario 2026"
- "Has Québec changed harassment rules recently?"
- "Find current CNESST guidance on psychological harassment"
- "What is the current minimum wage in Ontario?"
- "Current Canada Labour Code leave rules"
- "Recent HRTO decision about accommodation"

**Queries that never trigger web search:**

- Personal wellness queries ("I feel depressed", "What self-care strategies...")
- Crisis/self-harm messages
- Ambiguous or out-of-scope messages
- Document drafting requests (unless the user asks for current source verification)
- Simple stable HR questions answered from vetted internal guidance

**Routes where `webSearchAllowed` is always `false`:**

- `possible_crisis_or_self_harm`
- `personal_wellbeing`
- `personal_mental_health`
- `ambiguous`
- `out_of_scope`
- `document_drafting`

Crisis resources are maintained in `src/safety/crisisRouting.ts` from public Canadian government sources. The engine never uses Startpage to generate emergency resources dynamically.

### Web Source Validation

All candidate URLs returned by Startpage are validated before use:

- Non-http(s) URLs are suppressed.
- Localhost and private IP/internal URLs are rejected.
- Startpage anonymous/proxy URLs are rejected — they are not canonical citations.
- Tracking parameters are stripped from all URLs.
- Each URL is classified into a source type for authority ranking.

**Source type authority ranking (highest to lowest):**

| Source Type | Examples |
| --- | --- |
| `legislation` | `laws-lois.justice.gc.ca`, `ontario.ca/laws`, `legisquebec.gouv.qc.ca`, `canlii.org` |
| `official_government` | `canada.ca`, `ontario.ca`, `legisquebec.gouv.qc.ca` |
| `regulator_or_agency` | `cnesst.gouv.qc.ca`, `ohrc.on.ca`, `esdc.gc.ca`, `wsib.on.ca` |
| `court_or_tribunal` | `hrto.ca`, `chrc-ccdp.gc.ca`, `scc-csc.ca` |
| `reputable_secondary` | `hrreporter.com`, `hrpa.ca`, `fasken.com` |
| `general_web` | Everything else (returned as `requires_review`) |

**Web sources are not legal citations.** `route.legalBasisAllowed` controls legal citation rendering. `route.webSearchAllowed` controls web source rendering. They are independent gates.

### Privacy and Query Redaction

Before building a Startpage query, the engine **automatically redacts PII** from the user message:

- Employee and employer names
- Email addresses, phone numbers, SIN-like numbers, postal codes
- Street addresses, internal case/file numbers
- Medical diagnoses and sensitive personal details

**Example:**

User message: *"Jane Smith at Acme Manufacturing in Ottawa disclosed depression and asked for leave. What current Ontario guidance applies?"*

Search query becomes: *"site:ontario.ca Ontario mental health accommodation leave employment standards official guidance"*

Not: *"Jane Smith Acme Manufacturing depression leave"* or *"site:ontario.ca Ontario [MEDICAL\_CONDITION] accommodation leave"*

Key properties of the search query:

- Placeholders (`[MEDICAL_CONDITION]`, `[EMPLOYER]`, `[PERSON]`, etc.) are **never** sent to Startpage.
- Raw medical diagnoses (e.g. "depression") are **generalized** to safe legal topic terms (e.g. "mental health accommodation") before the query is built.
- The query preserves the **legal topic** (accommodation, leave, standards) and **jurisdiction** (Ontario), but not the sensitive workplace narrative.
- City names (Ottawa), employer names, and employee names are stripped.
- Repeated terms and stray punctuation (e.g. "duty to accommodate duty to accommodate", isolated "?") are deduplicated and removed before the query is sent.
- Temporal/meta noise words are stripped from the query: `changed`, `changes`, `year`, `years`, `updated`, `update` — these describe intent, not the legal topic.
- The word "current" used as a standalone adjective is stripped (it is noise for Startpage and adds no search value).
- Stray single letters left over from partial phrase stripping are removed from the final query.

The user's sensitive workplace narrative is never sent to Startpage.

### Web Search vs Internal Guidance Retrieval

The engine has three distinct knowledge sources that are never collapsed:

| Source | Type | Authority |
| --- | --- | --- |
| Deterministic playbooks | Static vetted guidance | High — editorial |
| Internal vetted guidance retrieval | Scored knowledge base | High — editorial |
| Startpage web search | Real-time discovery | Variable — validated per source |

Web search **supplements** internal guidance for current/recent external information. It does **not** override vetted internal guidance unless the engine explicitly flags possible outdated internal content. Qwen receives web search context only after deterministic routing, jurisdiction gating, and source validation. Qwen never performs its own browsing and never invents web citations.

### Current-Information Fallback Behaviour

When a user asks for **current, recent, or source-verified** information (e.g. "What changed in Ontario employment law this year?", "Latest ESA update Ontario 2026") and Startpage web search is **disabled or unavailable**, the engine behaves as follows:

**Broad current-update queries** (e.g. "What changed in Ontario employment law this year?") — no specific HR sub-topic:

- The engine does **not** claim to have verified current law.
- It does **not** serve unrelated internal static guidance (e.g. ESA termination minimums) as an answer to the "what changed this year" question.
- It returns a **bounded response** explaining that current-source verification is unavailable and provides safe next steps (check official sources, enable/configure web search).
- `route.retrievalAllowed` and `route.legalBasisAllowed` are set to `false` in the short-circuit response — the consuming app can rely on these gates without inspecting the workspace payload.
- `quality.blockedRendering` includes both `"retrieval"` and `"legalBasis"`.
- It adds a `quality.warnings` entry identifying why current verification was not performed.
- If the jurisdiction is known, the response acknowledges it without re-asking.
- If the jurisdiction is unknown, jurisdiction is still requested.

**Topic-specific current queries** (e.g. "What is the current minimum wage in Ontario?", "Find current CNESST guidance on harassment") — a specific HR sub-topic is present:

- The engine **does** short-circuit, just like broad queries, when web search is unavailable.
- Instead of returning unrelated internal guidance (e.g. termination/ESA minimums for a minimum wage question), it returns a **topic-aware bounded response** that acknowledges the specific topic, explains why current verification is unavailable, and points to the correct official source for that topic.
- `route.retrievalAllowed` and `route.legalBasisAllowed` are set to `false` — no internal guidance is returned as an answer to the current-data question.
- `quality.warnings` includes a retrieval-withheld entry so the consuming app can identify that internal guidance was suppressed.
- If the jurisdiction is known, the bounded response uses it (e.g. "in Ontario", "for federally regulated employers") without re-asking.

**Topic-alignment filtering for `workspace.retrievedGuidance`:** When web search is enabled and the pipeline proceeds normally (stable HR queries, not current-info), `workspace.retrievedGuidance` is filtered to only return items whose category matches the query topic. For example, a harassment investigation query will only receive harassment-category items — not termination, accommodation, or leave guidance that happened to score above the retrieval threshold. This prevents irrelevant guidance from appearing in the workspace payload.

**Consuming app guidance:** Always surface `quality.warnings` to the operator or user when a current-info request could not be verified. Do not render `workspace.retrievedGuidance` as an answer to a current-change question unless `route.webSearchAllowed=true` and web search returned validated sources.

**`quality.warnings` codes for current-info failures:**

| Warning text | Meaning |
| --- | --- |
| `Web search was requested but Startpage configuration is incomplete.` | `WEB_SEARCH_ENABLED=true` but `STARTPAGE_BASE_URL` or `STARTPAGE_API_KEY` is missing. |
| `Current-source verification was requested but web search is disabled (WEB_SEARCH_ENABLED=false).` | Global web search switch is off. |
| `Current-source verification was requested but web search was not enabled for this request (options.enableWebSearch=false).` | Per-request flag not set. |

### Web Search Security

- Startpage API keys are **never** logged, included in error messages, API responses, or debug payloads.
- The provider sanitizes all error messages before surfacing them as quality warnings.
- Debug details about failed web search appear only when `options.includeDebug=true`.
- If Startpage fails, the advisor continues with deterministic and vetted guidance and adds a quality warning.
